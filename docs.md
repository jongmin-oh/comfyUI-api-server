# ComfyUI 서버 구조를 SDAPI 방식으로 재구성하기 위한 목적 정의 및 개발 계획

## 1. 문서 목적

이 문서는 현재 ComfyUI의 **큐 기반 비동기 실행 구조**를, 사용 목적에 맞게 **SDAPI 스타일의 동기 요청-응답 구조**로 재구성하려는 목표를 정리하고, 그에 따른 개발 방향과 범위를 명확히 하기 위해 작성한다.

본 문서는 단순한 기술 검토가 아니라, 실제 구현을 전제로 한 **아키텍처 전환 기획 문서**이다. 따라서 다음 세 가지를 분명히 하는 데 목적이 있다.

* 왜 기존 ComfyUI 구조를 그대로 사용하는 것이 목적에 맞지 않는지
* 어떤 형태의 서버 구조를 최종 목표로 하는지
* 이를 어떤 단계로 개발할 것인지

---

## 2. 배경

ComfyUI는 본래 **워크플로우 실행 엔진 + UI 기반 오케스트레이션 도구**로 설계되어 있다. 따라서 외부 요청을 처리하는 방식도 다음과 같은 특성을 가진다.

* 요청을 즉시 처리하지 않고 큐에 적재함
* `prompt_id` 중심으로 작업을 추적함
* 실행 결과를 WebSocket, history 조회, 파일 조회 등의 방식으로 나중에 수집함
* 사용자 또는 외부 클라이언트가 작업 완료를 별도로 확인해야 함

이 구조는 UI 중심 사용성과 워크플로우 유연성에는 적합하지만, 다음과 같은 목적에는 불리하다.

* 일반적인 REST API 서버처럼 즉시 응답하는 인터페이스 제공
* 외부 서비스에서 `txt2img`, `img2img` 같은 요청을 단일 호출로 처리
* AUTOMATIC1111의 sdapi와 유사한 사용성을 기대하는 클라이언트 호환
* 결과를 나중에 polling 하거나 WebSocket으로 수신하지 않고, 요청 1회로 생성 결과를 받는 구조

즉, 현재 문제는 단순히 FastAPI를 붙일지 말지의 문제가 아니라, **ComfyUI의 동작 철학 자체가 현재 목표와 맞지 않는다는 점**에 있다.

## 3. 추진 목적

본 프로젝트의 핵심 목적은 다음과 같다.

### 3.1 외부 인터페이스를 SDAPI 방식으로 통일

클라이언트가 ComfyUI의 내부 개념인 queue, prompt_id, history, websocket을 알 필요 없이, 아래와 같은 방식으로 사용할 수 있도록 한다.

* 요청 전송
* 서버 내부에서 즉시 생성 수행
* 응답에서 결과 이미지 또는 결과 메타데이터를 직접 반환

즉, 사용자 입장에서 서버를 **작업 큐 시스템**이 아니라 **동기식 이미지 생성 API 서버**로 인식하게 만드는 것이 목적이다.

### 3.2 UI와 API의 실행 기반을 하나로 통합

별도의 프록시 서버를 앞단에 두는 방식이 아니라, **ComfyUI 동작과 API 동작을 동일한 런타임/동일한 프로세스 안에서 통합 운영**하는 것을 목표로 한다.

이를 통해 다음을 기대한다.

* 배포 구조 단순화
* 프로세스 간 통신 제거
* 내부 상태 공유 단순화
* API와 UI가 동일한 모델/리소스 상태를 직접 사용

### 3.3 큐 기반 사용자 경험 제거

현 구조에서는 요청 후 결과를 다시 조회해야 하는 흐름이 필수적이다. 그러나 본 프로젝트는 이를 제거하고, **요청-응답 한 번으로 작업을 완료하는 사용자 경험**을 제공하는 데 목적이 있다.

### 3.4 향후 API 제품화 기반 확보

최종적으로는 ComfyUI를 단순 워크플로우 툴이 아니라, **서비스 내장형 생성 API 엔진**으로 재구성하는 기반을 만드는 것을 목표로 한다.

즉, 본 작업은 단기적으로는 sdapi 호환성을 위한 것이지만, 장기적으로는 아래 방향으로 이어질 수 있다.

* 내부 서비스용 이미지 생성 백엔드
* SaaS/API 제품형 생성 서버
* 워크플로우 기반이지만 API 중심인 inference backend

## 4. 내가 원하는 최종 상태

내가 원하는 최종 구조는 단순한 “ComfyUI 앞단 래퍼”가 아니다. 원하는 형태는 다음과 같다.

### 4.1 서버 형태

* ComfyUI와 API 서버가 분리되지 않음
* 하나의 실행 프로세스 안에서 UI와 API가 함께 동작함
* API 엔드포인트는 sdapi 스타일을 따름

예시:

* `POST /sdapi/v1/txt2img`
* `POST /sdapi/v1/img2img`
* `GET /sdapi/v1/options`
* `GET /sdapi/v1/samplers`

### 4.2 요청 처리 방식

* 클라이언트 요청을 받으면 즉시 처리 시작
* 내부에서 Comfy workflow를 생성 또는 매핑
* 실행 완료까지 서버가 내부적으로 대기
* 완료 시 결과 이미지와 메타데이터를 응답으로 반환

즉, 외부에서 볼 때는 **완전한 동기식 REST API**처럼 보여야 한다.

### 4.3 내부 엔진 활용 방식

여기서 중요한 점은 “외부 구조를 sdapi처럼 바꾸는 것”이지, 반드시 엔진 내부의 모든 큐 메커니즘을 물리적으로 삭제하는 것이 목표는 아니라는 점이다.

실제로는 다음 둘 중 더 안정적인 구조를 선택할 수 있다.

* 내부적으로는 큐/실행기 구조를 유지하되, API 계층에서 완료까지 await
* 또는 일부 executor 경로를 직접 호출하는 동기 실행 경로를 별도로 설계

다만 중요한 것은 **외부 인터페이스가 sdapi처럼 보이는 것**이며, 사용자는 queue/history/websocket을 의식하지 않아야 한다.

### 4.4 결과 반환 방식

응답은 sdapi 사용성에 맞게 다음을 지원해야 한다.

* base64 이미지 반환
* seed, sampler, steps, cfg, width, height 등 메타데이터 반환
* 필요 시 infotext 또는 generation parameters 반환
* 오류 발생 시 명확한 HTTP 에러 코드와 메시지 반환

## 5. 범위 정의

### 5.1 포함 범위

본 프로젝트에서 포함하는 범위는 다음과 같다.

* ComfyUI 내부 프로세스에 API 엔드포인트 통합
* sdapi 호환 입력 스키마 설계
* sdapi 요청값을 Comfy workflow로 매핑하는 계층 구현
* 실행 완료 대기 및 결과 직렬화 구현
* 에러 처리 및 타임아웃 처리
* 결과 이미지 반환 형식 통일
* 기본적인 옵션 조회용 엔드포인트 설계

### 5.2 제외 범위

초기 단계에서 제외할 수 있는 범위는 다음과 같다.

* AUTOMATIC1111의 모든 확장 기능 완전 호환
* 모든 커스텀 노드 조합에 대한 자동 변환
* Web UI 전체를 sdapi 중심 UX로 재설계
* 멀티테넌시/권한 관리/과금 시스템
* 분산 큐/멀티 GPU 스케줄링 고도화

즉, 1차 목표는 **완전한 A1111 복제품**이 아니라, **ComfyUI 엔진 기반의 sdapi 스타일 서버**를 만드는 것이다.

## 6. 핵심 설계 원칙

### 6.1 외부 API는 단순해야 한다

외부 사용자는 workflow JSON, node graph, prompt_id 같은 개념을 몰라도 된다. API는 가능한 한 이미지 생성 서버 관점의 평면적인 파라미터 구조를 제공해야 한다.

### 6.2 내부 엔진은 최대한 재사용한다

ComfyUI의 강점은 워크플로우 엔진과 노드 생태계에 있다. 따라서 이를 버리기보다, 상위 API 계층만 다시 설계하는 방식이 유지보수 측면에서 유리하다.

### 6.3 UI와 API의 공존을 깨지 않는다

이 프로젝트는 UI를 버리고 API만 남기는 작업이 아니라, **기존 UI 사용성은 유지하면서 API 인터페이스만 재정의**하는 방향을 우선 고려한다.

### 6.4 동기 응답 구조를 우선한다

비동기 큐 추적 방식은 내부 구현에만 남고, 외부에는 가능한 한 요청-응답 완료형 구조를 제공한다.

### 6.5 점진적으로 확장 가능해야 한다

처음에는 `txt2img`, `img2img` 같은 핵심 기능부터 구현하고, 이후 ControlNet, upscaler, inpaint, LoRA 옵션 등으로 확장 가능해야 한다.

## 7. 개발 전략

### 7.1 1단계: 최소 동작 구조 확보

목표는 가장 기본적인 `txt2img` 동기 응답 API를 ComfyUI 내부에 구현하는 것이다.

주요 작업:

* API 라우트 추가
* 입력 파라미터 정의
* 기본 workflow 템플릿 설계
* 요청값을 workflow에 매핑
* 실행 완료까지 대기
* 결과 이미지 반환

이 단계의 산출물:

* `POST /sdapi/v1/txt2img` 동작
* base64 결과 반환
* 기본 메타데이터 반환

### 7.2 2단계: 공통 실행 레이어 정리

단일 엔드포인트만으로는 유지보수가 어렵기 때문에, 실행 레이어를 공통 모듈로 분리한다.

주요 작업:

* workflow builder 분리
* executor wrapper 분리
* result serializer 분리
* 예외 처리 구조화
* timeout/cancel 처리 추가

이 단계의 목적은 **라우트 추가가 쉬운 구조**를 만드는 것이다.

### 7.3 3단계: img2img 및 옵션 엔드포인트 확장

주요 작업:

* `POST /sdapi/v1/img2img`
* `GET /sdapi/v1/samplers`
* `GET /sdapi/v1/options`
* 모델 정보/기본 설정 조회

이 단계부터는 sdapi 클라이언트와의 실질적인 호환성이 높아진다.

### 7.4 4단계: 고급 기능 대응

필요에 따라 다음을 순차 추가한다.

* ControlNet 매핑
* LoRA 로딩/선택
* 업스케일링
* Inpaint
* Batch 생성
* Progress API
* Interrupt/Cancel API

이 단계는 기본 구조가 안정화된 후 진행해야 한다.

## 8. 권장 아키텍처

### 8.1 권장 구조 개요

권장하는 구조는 아래와 같다.

1. **API Route Layer**

   * sdapi 스타일 HTTP 엔드포인트 제공
   * 요청 검증 및 응답 형식 통일

2. **Parameter Mapper**

   * sdapi 요청값을 내부 workflow 파라미터로 변환
   * sampler, seed, size, prompt 등 매핑

3. **Workflow Builder**

   * 템플릿 workflow 생성 또는 수정
   * txt2img/img2img별 그래프 구성

4. **Execution Wrapper**

   * ComfyUI 내부 실행 엔진 호출
   * 완료까지 대기
   * timeout 및 예외 처리

5. **Response Serializer**

   * 이미지 결과를 base64로 인코딩
   * 메타데이터를 sdapi 형식으로 직렬화

### 8.2 왜 이 구조가 필요한가

이 구조가 필요한 이유는 라우트 코드에 workflow 생성, 실행, 결과 반환 로직이 섞이면 금방 유지보수가 어려워지기 때문이다.

특히 이후 기능이 늘어나면 다음 문제가 생긴다.

* 엔드포인트마다 workflow 로직 중복
* 예외 처리 중복
* 응답 형식 불일치
* 커스텀 기능 추가 시 영향 범위 확대

따라서 처음부터 계층을 나누는 것이 좋다.

## 9. 예상되는 기술적 난점

### 9.1 ComfyUI 본래 구조와의 충돌

ComfyUI는 본래 큐와 이벤트 기반 흐름을 전제로 설계되어 있기 때문에, 동기 응답 구조를 덧씌울 때 내부 상태 관리와 충돌 가능성이 있다.

### 9.2 업스트림 업데이트 추적 문제

핵심 서버 로직에 손을 많이 대면 ComfyUI 최신 버전을 따라가기 어려워질 수 있다. 따라서 가능한 한 변경 지점을 명확히 분리해야 한다.

### 9.3 워크플로우 범용성 문제

ComfyUI는 매우 자유로운 graph 기반 구조이고, sdapi는 비교적 고정된 파라미터 기반 구조이다. 따라서 모든 workflow를 일반화된 sdapi 파라미터로 표현하기는 어렵다.

### 9.4 에러 처리 복잡성

노드 실행 오류, 모델 로딩 실패, 잘못된 sampler 지정, 이미지 디코딩 문제 등 다양한 예외를 HTTP 응답 수준으로 일관되게 변환해야 한다.

### 9.5 성능에 대한 오해

구조를 sdapi처럼 바꾼다고 해서 생성 속도 자체가 극적으로 빨라지는 것은 아니다. 실제 병목은 모델 로딩, VRAM, sampler, workflow 복잡도 등에 있을 수 있다.

즉, 본 프로젝트는 **주로 인터페이스와 운영 구조 개선**이며, 성능 최적화는 별도 과제로 병행해야 한다.

## 10. 성공 기준

본 프로젝트가 성공했다고 판단할 기준은 다음과 같다.

* 외부 클라이언트가 queue/history/websocket 없이 API를 사용할 수 있다.
* `txt2img` 요청 1회로 결과 이미지가 직접 반환된다.
* ComfyUI UI와 API가 한 프로세스 내에서 함께 동작한다.
* ComfyUI 내부 엔진을 활용하면서도 sdapi 스타일 응답을 제공한다.
* 추후 `img2img`, ControlNet, LoRA 등으로 확장 가능한 구조를 가진다.
* 코드 구조가 라우트/매핑/실행/직렬화 계층으로 분리되어 유지보수 가능하다.

## 11. 최종 방향 정리

내가 원하는 것은 단순한 프록시나 외부 래퍼가 아니다.

나는 다음을 원한다.

* ComfyUI와 FastAPI 동작을 하나의 서버 구조로 통합하고 싶다.
* 외부에는 sdapi 스타일의 동기 API를 제공하고 싶다.
* queue, prompt_id, websocket, history 조회를 외부 인터페이스에서 제거하고 싶다.
* 내부적으로는 ComfyUI의 엔진과 워크플로우 강점을 최대한 활용하고 싶다.
* 결과적으로 ComfyUI를 UI 중심 툴이 아니라 **API 중심 생성 서버**로 재구성하고 싶다.

즉, 이 프로젝트의 본질은 기능 추가가 아니라 **서버 성격 자체를 전환하는 아키텍처 재정의 작업**이다.

---

## 12. 한 줄 요약

**ComfyUI를 그대로 감싸는 것이 아니라, ComfyUI 내부 실행 엔진을 유지한 채 외부 인터페이스와 서버 계층을 SDAPI 중심으로 재구성하여, UI와 API가 통합된 동기식 생성 서버로 전환하는 것이 본 프로젝트의 목표다.**
